Claude Code 不是一个只会补全代码的编辑器插件，它更像一个运行在终端里的协作型开发助手。它能读取项目文件、执行命令、修改代码、解释架构，也能围绕一个目标连续推进多步任务。

如果你刚开始用 Claude Code，最重要的不是一上来就让它重构整个项目，而是先学会三件事：让它理解项目、把任务说清楚、在关键节点检查它的判断。

## 适合解决什么问题

Claude Code 最适合处理“需要理解上下文”的开发任务，比如：

- 接手一个陌生项目，快速梳理目录结构和启动方式。
- 修复一个能复现的 bug，并补上测试。
- 做小范围重构，同时保持原有接口不变。
- 根据已有代码风格新增页面、接口或脚本。
- 帮你 review 当前改动，找出风险和遗漏。

如果问题只是查一个 API 怎么用，普通聊天工具也够；如果问题需要读仓库、跑命令、改文件，Claude Code 的优势就明显了。

## 安装与启动

官方推荐直接安装 Claude Code CLI。不同系统命令略有区别，安装完成后在项目目录运行：

```bash
claude
```

如果你想让 Claude Code 从某个项目根目录开始理解上下文，先进入项目目录：

```bash
cd your-project
claude
```

第一次进入后，可以先让它只做分析，不改代码：

```text
请先阅读这个项目的结构，告诉我它的技术栈、启动命令、主要目录职责，不要修改文件。
```

这个开场很重要。你给它的第一条任务决定了它是“先理解”还是“直接动手”。对陌生项目，先读再做通常更稳。

## 常用斜杠命令

Claude Code 有一批斜杠命令，用来控制会话和工作流。新手最常用的是：

```text
/help
/clear
/compact
/model
/skills
/hooks
```

`/help` 用来查看当前可用能力；`/clear` 适合彻底开启新任务；`/compact` 适合长会话中压缩上下文，保留关键结论；`/model` 用于切换模型；`/skills` 和 `/hooks` 则对应扩展能力。

一个实用习惯是：每完成一个阶段，让 Claude Code 总结当前状态，再继续下一阶段。

```text
请总结目前你已经确认的事实、做过的修改、还没验证的风险。先不要继续改代码。
```

这能避免长会话里上下文逐渐变乱。

## 第一个真实任务怎么发

差的任务通常是这样的：

```text
帮我优化这个项目。
```

Claude Code 也许会动起来，但它不知道你真正想要的是性能、架构、样式、类型，还是构建速度。

更好的任务应该包含目标、边界和验收方式：

```text
请帮我修复登录页在移动端按钮换行的问题。
要求：
1. 只修改登录页相关组件和样式。
2. 保持现有颜色和设计风格。
3. 修改后运行现有前端检查。
4. 完成后告诉我改了哪些文件，以及如何验证。
```

这个提示把“要做什么”和“不要做什么”都说清楚了。Claude Code 越少猜测，结果越接近你的预期。

## 推荐的新手工作流

我建议按下面这个节奏开始：

1. 让 Claude Code 只读项目，输出结构理解。
2. 让它提出解决方案，不立即写代码。
3. 你确认方向后，再让它小步修改。
4. 每次修改后运行测试、构建或最小验证命令。
5. 最后让它总结改动和残留风险。

对应的提示可以这样写：

```text
先不要修改文件。请阅读相关代码，给出你打算怎么修，以及会影响哪些文件。
```

确认后再说：

```text
按刚才的方案实现。保持改动尽量小，完成后运行验证命令。
```

## 注意事项

不要把密钥直接发进会话，也不要让 Claude Code 把 `.env` 提交到仓库。涉及部署、数据库、支付、线上数据时，要让它先解释风险和回滚方式。

Claude Code 很擅长执行，但你仍然是最后的技术负责人。它给出的修改要经过测试，涉及业务逻辑的地方尤其要看清楚。
